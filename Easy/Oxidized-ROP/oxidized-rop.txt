allowing the attacker to change a variable's value and ultimately gain code execution.

In Rust, the last expression of a function is implicitly used as the return value of that function. So,
read_user_input() returns a buffer containing our input, which has been trimmed of the

newline characters at the end.

This function takes two arguments; a mutable 8-bit unsigned integer array, dest , and a String
pointer, src . First, it checks that the count of the characters in the src argument is less than 200.
Then using an unsafe code block, the characters are copied to the destination using a destination
pointer.

If the pin is not equal to 123456 , it immediately returns. In the case that it is, /bin/sh is called
and we have a shell. We have to find a way to alter the pin to be 123456 in order to solve the
challenge.

Exploitation
The unsafe code block hints that there is a buffer overflow present here. Researching what
.chars().count() does and what's the difference between that and len() , we quickly find out

that len() returns the number of bytes in a String, whereas .chars().count() returns the
number of characters in a String. In a language like C these two things would be equal, but Rust
treats characters a little differently. Characters in Rust are 4 bytes long to support all unicode
characters.
But the array we declared to store our statement stores 8-bit unsigned integer values, so we
clearly have a buffer overflow.
Since the pin is mutable and is initialized after our Feedback structure, it most likely sits after our
buffer in virtual memory, and we can overflow it using the Complete Survey option.
Let's run the binary with gdb , and check if our hypothesis is correct.
